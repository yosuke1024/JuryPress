# System Instructions for AI Agent: Architecture & Coding Conventions

You are an autonomous, self-verifying AI Developer Agent. When developing and refactoring this application, you MUST strictly adhere to the following principles to minimize blast radius, optimize token usage, and ensure long-term maintainability.

> [!WARNING]
> **CRITICAL EXECUTION MANDATE**: 
> You MUST NEVER commit directly to `main` or modify files on `main` without creating a branch first. 
> Every single coding task MUST start with `git checkout -b feature/...` or `fix/...`. 
> Upon completion, you MUST push the branch and create a Pull Request.

Each rule includes applicable conditions and exceptions. Do not blindly apply them if the context does not fit.

## 1. Core Architecture (アーキテクチャ原則)
* **Vertical Slice Architecture & Shared Infra:** UI、ロジック、データアクセスは基本的に機能（Feature）単位で同一ディレクトリにカプセル化せよ。ただし、DB接続、認証、エラーハンドリングのような共通インフラ、および汎用的で抽象化されたUIコンポーネントのみ `shared/` や `infra/` などの共通層での横断的ファイル分割を明示的に許容する。
* **Rule of Three / WET over DRY:** コンポーネント間の過度な結合による「依存関係の迷子」を避けるため、2回までのコード重複（WET）は積極的に許容する。しかし、「3箇所以上での完全な重複」が発生した場合のみ、システム的な保守性を考慮し共通化（DRY）を検討すること。
* **Single Source of Truth:** API仕様（OpenAPI/GraphQL）やDBスキーマを唯一の情報源とし、型定義やバリデーションロジックはそこから自動生成して使用せよ。

## 2. API & State Management (通信と状態管理)
* **BFA vs Local State Sync:** 以下の基準で使い分けよ。
  * **BFA (Backend For Agent) Pattern:** 実行時に最新状態が必要なデータや、複数リソースにまたがる複雑な操作は、N+1問題を防ぐため集約エンドポイント（BFF）を利用して1回の通信で完結させよ。
  * **Local State Sync:** 都道府県マスタやシステム設定など、頻繁に参照されるが更新頻度が低いデータは都度APIを叩く過剰設計を避け、ローカル環境（SQLite等）に同期・キャッシュしてからクエリを実行せよ。
* **Bulk Operations First:** 複数データの作成・更新処理はループで個別APIを叩くのではなく、必ず配列を受け取るバルク処理APIを利用・設計せよ。
* **Event-Driven with Polling Fallback:** 長時間の待機状態では、可能ならWebSocket/SSE等のイベント駆動を利用する。しかしエージェントの実行環境（CLI環境やServerless）により常時接続が困難な場合は、指数バックオフを伴うセーフ・ポーリングを限定的なフォールバックとして許容する。

## 3. Coding Standards (実装規約)
* **Strict Public Typing & Local Inference:** パブリックなインターフェース（関数の引数・戻り値、APIレスポンス、Props等）には、厳密な静的型付け（TS/Go）を必須とする（`any` や過度な `unknown` の乱用は厳禁）。一方、関数内のローカル変数は、トークン節約と可読性向上のため、安全な型推論を積極的に利用せよ。
* **Explicit Error Handling:** エラーの握り潰し（サイレントフェイル）は厳禁。エラーは後述の構造化ログ（Trace ID付き）で出力し、エージェント自身でパースしてRoot Causeを自律特定・修復可能にせよ。
* **Strict i18n & Localization (ハードコードの禁止):** 画面に表示されるテキスト（UIラベル、例外メッセージ、プレースホルダー等）に対して、コンポーネント内で直接 `locale === "ja" ? '日本語' : '英語'` のような三項演算子を用いた簡易的なローカライズを記述してはならない。必ず `src/locales/index.ts` および `supplemental.ts` に翻訳キーセット（ja, en, th, fi, id, vi, ms, hi）を追加し、`useLanguage().t("key")` を用いて参照すること。AIによる自動翻訳で構わないため、全サポート言語のキーを埋めること。

## 4. Self-Verification (自己検証プロセス)
* **Targeted Testable UI (`data-testid`):** 全てのUIではなく、「ユーザーが操作する、またはAgentが検証対象とするインタラクティヴ要素」に限定して `data-testid` を付与せよ。リスト要素などの動的項目には必ず `data-testid="todo-item-${id}"` のような一意の命名規則を用いること。
* **Atomic Steps:** 1つの機能追加・修正ごとに、必ずブラウザ操作や単体テストによる動作検証を行い、成功を確認してから次のステップ（機能実装）に進むこと。

## 5. Security & Environment (セキュリティと環境設定)
* **Strict Env & Secrets Masking:** 認証キー等は環境変数（`.env`）から読み込み、アプリ起動時にZod等のスキーマバリデーションを実行せよ（Fail-Fast）。また、`.env` のコミット禁止（`.gitignore`）はもちろん、エラーログ出力プロセスにおける機密情報のマスキングを徹底せよ。
* **Injection & Dynamic Execution Prevention:** DBクエリやシェル呼び出しを実行する際は、プリペアードステートメントやサニタイズを徹底せよ。また、悪意あるユーザー入力による任意のコード実行を防ぐため、`eval`、`new Function`、および入力値に依存する動的 `import()` は原則禁止とする。
* **Security Checks:** Agentは、新パッケージの導入時に依存ライブラリの脆弱性チェック（`npm audit` 等）を自発的に行う責務を持つ。

## 6. Performance & Protective Design (防御的パフォーマンス設計)
* **Fail-Safe Pagination:** レスポンスが極小の想定でも、DBクエリやAPI取得には必ず「デフォルト上限（Limit）」または「ページネーション」を義務付ける。ただし、システム設定値（単一レコード等）のように要素数が静的かつ自明な場合は例外として全件取得を許容する。

## 7. Observability (可観測性とデバッグ容易性)
* **Traceable Context Logging:** エラー単体ではなく「処理の開始から完了まで」を追跡可能な「Trace ID（または Correlation ID）」を付与し、構造化ログ（JSON等）として出力せよ。これにより、Agentのデバッグ作業が「推測」からログベースの「事実のトラッキング」へと変わる。

## 8. Agent Collaboration Rules (AI行動ルールと例外基準)
* **Document the "Why" over "What":** 特殊な仕様、ハック、または通常のセオリーから外れた実装（上記の例外事項を含む）には、必ず「なぜ（Why）そうしたか」の背景をJSDoc等で直書きせよ。Agentが将来「不要なコードだ」と誤認識して消去・破壊するのを防ぐためである。
* **Limit Blast Radius (スコープ外改修の禁止と例外):** タスク対象外のコードで古い実装や気付いた改善点があっても「ついで修正」は厳禁。ただし、**「シグネチャ変更に伴う呼び出し元の修正」といった、タスクに直接起因する波及的変更は『例外（必ず対応すべき修正）』とする**。これを放置（Agentの修正見落とし）してはならない。影響範囲が想定以上に広くなる場合は、事前にPlan（計画）を作成しUserの承認を得よ。
* **Documentation Updates:** 新機能の追加や仕様変更を伴うコーディングタスクを完了した際は、実装コードだけでなく `README.md` をはじめとする使い方ガイド・ドキュメント類のアップデートも自発的に行うこと。ドキュメントの更新忘れは、将来のシステム拡張や別エージェントがアサインされた際の環境理解において、重大な「実装との乖離リスク」となる。

## 9. Spec-Driven Implementation & ADR (仕様駆動の実装と履歴の一元化)
本プロジェクトでは、仕様策定エージェントと実装エージェントが異なる場合がある（例: Opusが仕様を書き、Geminiが実装する）。チャットのコンテキスト断絶を防ぎ、かつ仕様と実装履歴を同一ファイルでSingle Source of Truthとして維持するため、以下のライフサイクルを厳守すること。

* **Single Source of Truth:** 実装に着手する際、エージェントはチャット履歴の断片的な会話に依存せず、常に `docs/current/` 内の決定済みMarkdown仕様書を唯一の情報源（正）として扱うこと。`docs/archive/` や `.gemini/antigravity/brain/` 内の過去の会話アーティファクトは**参照禁止**。
* **Auto-Discovery & Read Before Code:** ユーザーから抽象的な実装指示を受けた場合、コードを書く前に必ず `docs/current/` ディレクトリ内をスキャンし、該当仕様書を読み込んで要件を完全に把握せよ。
* **Spec Status (YAML Frontmatter):** 仕様書のフロントマターには必ず `status: draft | approved | implemented` を記載する。エージェントは `approved` のもののみ実装を開始でき、`draft` の場合はユーザーに承認の可否を確認すること。
* **Lifecycle & Implementation Report (ADR):** ファイルの移動（リンク切れの原因）や、別ディレクトリ（history等）への履歴ファイルの二重作成は厳禁とする。実装を完了したAIエージェントは、以下のフローで仕様書自体を「完了記録（ADR）」へ昇格させること。
  1. フロントマターを `status: implemented` に更新する。
  2. 当該仕様書ファイルの末尾に `## Implementation Report` セクションを追記し、「実装時の変更点」「苦労した点・非自明な回避策」「なぜその設計を採用したか（Why）」を記録する。
  3. `docs/README.md` のインデックスを更新する。
* **Spec vs Code Conflict:** 実装中に仕様書と既存コードが矛盾する場合、**仕様書を正とする**。技術的な理由で仕様通りに作れない、または重大なリスクが見つかった場合は、独断でコードを変更せずユーザーに確認を仰ぐこと。
* **🔄 SSOT維持ルール（会話中の差分検知と更新）:** 会話中にユーザーの発言が `docs/current/` の内容と矛盾・乖離していると気づいた場合、以下のフローを厳守する：
  1. **即座にユーザーに確認する。**「`docs/current/` には○○と記載されていますが、今の方針は△△でしょうか？」と明示的に問う。
  2. **ユーザーの承認を得てから、`docs/current/` の該当ファイルを更新する。**
  3. **AIが独断で `docs/current/` を書き換えることは禁止。** 必ずユーザーの明示的な承認が先。
  4. 更新時は、ファイル末尾の `最終更新` 日付も合わせて更新すること。
## 10. Product Principles (プロダクト基本指針の遵守)
* **Mandatory Read:** 新機能の追加、仕様変更、マネタイズに関わる実装を行う前に、必ず `docs/PRODUCT_PRINCIPLES.md` を読み、その指針に反していないことを確認せよ。
* **Feature Gate Check:** 新機能を追加する際は、指針内の「機能追加時の判断基準」（5つの質問）にすべて「はい」と答えられることを確認してから着手すること。答えられない項目がある場合は、ユーザーに確認を仰ぐこと。
* **Zero-Cost Free Tier:** 無料ユーザー（広告視聴なし）のアクションで外部API原価が発生するコードを書くことは厳禁。広告視聴ユーザーについては、**広告収益がそのAPIリクエスト1回あたりの原価を上回る場合に限り許可**する。判定は `docs/PRODUCT_PRINCIPLES.md` セクション2-2の原価テーブルに基づくこと。
* **Inclusive Positioning:** UIテキスト・ストア掲載文において、ターゲットを「スイーツ」や特定の料理ジャンルに限定する表現を使わないこと。「手作り料理」「フードビジネス」など包括的な表現を用いること。

## 11. Git / DevOps Workflow (CI/CDとリリース管理)
* **GitHub Flow:** 開発は常に対象機能ごとの `feature/*` または `fix/*` ブランチを作成して行い、直接 `main` ブランチにコミットしてはならない。実装終了後、PR（Pull Request）を作成し、自動テスト（CI）を通過した上で `main` へマージすること。
* **Automated PR Creation by AI:** AIエージェントが変更を `git push` した際は、`gh pr create` コマンドを使用してPull Requestを作成すること。
* **CI Verification Required:** `.github/workflows/ci.yml` にて `npm run build` によるリントエラー・TypeScript型チェック・静的書き出し機能の自動チェックが行われるため、コード実装時はコンパイルエラーを0に保たなければならない。
* **Clean the Workspace:** 古い使わなくなったテストスクリプトやルート直下の不要な一時ファイル等は、コミット前に自主的に削除し作業ツリーを常にクリーンに保つこと。

## 12. LLM Prompt Engineering Principles (AIプロンプト設計原則)
アプリ内でLLM（Gemini等）を呼び出すプロンプトを作成・修正する際は、ユーザーの指示を超えて、以下の原則を常に自発的に適用し、**効率的・効果的で、利用者に驚きと喜びを与える** 応答品質を追求すること。

* **Regional Culinary Intelligence (地域料理知識の活用):** GPS座標や写真の背景から地域を特定し、その土地特有の食材・調味料・油脂・調理法・ポーションサイズの知識を用いて栄養推定の精度を最大化せよ。同じ「カレー」でも北インド（ギー使用）と日本（小麦粉ルー）では脂質構成が全く異なる。
* **Sensory & Situational Feedback (五感的・状況的フィードバック):** 単にデータを返すのではなく、写真から読み取れる色彩・質感・鮮度、ユーザーの時間帯・場所・シチュエーションへの共感を言語化し、「AIが本当に自分を見てくれている」というライブ感を演出せよ。
* **Warm Coach Tone (温かいコーチのトーン):** ターゲットユーザーの継続利用を促進するため、罪悪感や批判を一切使わず、「優しい友人のような管理栄養士」のトーンを徹底する。重大な健康リスク（過度な塩分等）のみ、穏やかに注意喚起する。
* **Contextual Personalization (文脈に基づくパーソナライズ):** `aiDigest`（直近の食事傾向サマリー）をプロンプトに含め、単発ではなく「習慣の変化」に気づいて褒め、継続のモチベーションを高めるアドバイスを生成せよ。
* **Micro-Actionable Tips (すぐ実行できる小さな提案):** 「タンパク質を増やしましょう」のような抽象的助言ではなく、「次の食事にゆで卵を1つ追加するだけで完璧！」のように、具体的で即実行可能な提案を1つだけ含めよ。
* **Structured Prompts (構造化プロンプト):** XMLタグや明確なセクション分け（`<analysis_instructions>`, `【SECTION】`等）を用いて指示とデータを分離し、LLMの理解精度を向上させつつトークン効率を最適化せよ。
* **Token Efficiency (トークン効率):** プロンプトの冗長さを排除しつつ情報密度を保つ。固定テキストは最小限、変数で動的に注入するデータは必要十分な量に留めよ。追加コンテキスト（aiDigestなど）は~200トークン以内の固定サイズに設計すること。
